# Changelog - DJ CRM v2

Все заметные изменения в этом проекте будут документироваться в этом файле.

## [2.0.0] - 2024-05-02

### Добавлено

#### Модели
- **Core App**:
  - `Client` - модель клиента с полями: имя, фамилия, отчество, пол, дата рождения, телефон, email, адрес, родитель, статус, теги
  - `ContactPerson` - дополнительные контакты для клиента (второй родитель)
  - `EmployeeProfile` - профиль сотрудника (преподавателя/менеджера)
  - `Role` - система ролей с правами доступа (OWNER, ADMIN, MANAGER, TEACHER, STUDENT)
  - `RoleAssignment` - назначение ролей пользователям

- **Interactions App**:
  - `InteractionType` - типы взаимодействий (звонок, встреча, сообщение)
  - `Interaction` - история взаимодействий с клиентом
  - `Comment` - комментарии к взаимодействиям

- **Lessons App**:
  - `LessonType` - типы занятий (фортепиано, гитара, вокал)
  - `Lesson` - модель занятия со статусами, преподавателем и учеником
  - `Attendance` - посещаемость занятий

- **Payments App**:
  - `PaymentMethod` - способы оплаты (наличные, карта, перевод)
  - `PaymentPackage` - пакеты уроков для продажи
  - `Contract` - договор с клиентом
  - `Payment` - оплата (платежи)
  - `Invoice` - счета на оплату

- **Tasks App**:
  - `Task` - задачи для сотрудников с приоритетами и статусами

#### Сигналы (Автоматизация)
- **Core**:
  - Автообновление `silent_days` при изменении `last_interaction_date`
  - Автообновление статуса клиента при создании взаимодействия
  - Автоматическое создание `EmployeeProfile` при создании пользователя

- **Interactions**:
  - Автообновление `last_interaction_date` при создании взаимодействия
  - Автообновление статуса клиента при создании взаимодействия
  - Автоустановка текущей даты при создании взаимодействия

- **Lessons**:
  - Авторасчет времени окончания занятия на основе `LessonType.duration_default`
  - Авто создание записи посещаемости при создании занятия
  - Проверка на конфликты расписания

- **Payments**:
  - Автогенерация номера договора при создании
  - Автогенерация номера счета при создании
  - Автообновление статуса платежа при полной оплате
  - Авто создание платежа при создании счета

- **Tasks**:
  - Автоустановка `completed_at` при смене статуса на DONE
  - Проверка просроченных задач

#### Права доступа
- `core/permissions.py` - декораторы для проверки прав:
  - `@role_required` - проверка наличия роли
  - `@permission_required` - проверка наличия права
  - `@manager_required` - доступ менеджеров и выше
  - `@teacher_required` - доступ преподавателей и выше
  - `@finance_access_required` - доступ к финансовым данным
  - `@edit_finance_required` - редактирование финансовых данных

- `core/middleware.py` - middleware для проверки прав доступа

#### Админка
- Полная регистрация всех моделей в Django Admin
- Кастомные списки, фильтры и поиск для всех моделей
- Группировка полей через `fieldsets`

#### Утилиты
- `core/management/commands/init_roles.py` - команда для инициализации базовых ролей
- `requirements.txt` - список зависимостей
- `.env.example` - пример файла окружения
- `README.md` - документация проекта

### Изменено
- Полная переработка архитектуры приложений:
  - Старое: `products`, `sales`, `clients`
  - Новое: `core`, `interactions`, `lessons`, `payments`, `tasks`

- Обновлены модели `Client` с добавлением:
  - Статусы (ACTIVE, SILENT, LOST, CANCELLED, POTENTIAL)
  - Поле `silent_days` для отслеживания дней без активности
  - Поле `last_interaction_date` для автообновления статуса

- Обновлена модель `EmployeeProfile` с добавлением:
  - Связь с `Role` через `primary_role`
  - Поля `is_active`, `is_on_leave`, `leave_end_date`

### Удалено
- Старые приложения `products` и `sales` (заменены на `payments`)
- Устаревшая структура данных

### Технические детали
- Django 6.0.4
- Python 3.12
- PostgreSQL (с fallback на SQLite для разработки)
- Полная поддержка русского языка
- Часовой пояс: Asia/Yekaterinburg
